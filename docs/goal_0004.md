# Goal 0004: Improve Zig `insert_reserved` Performance

Optimize the Zig `zig_phmap` reserved-insert hot path until it is at least on-par with C++ `parallel-hashmap` on the same benchmark workload on the local development host.

## Background

Goal 0003 made the Zig port competitive or faster than C++ on most hash-map workloads, including lookup hits, iteration, remove, and string-key operations. Goal 0004 originally investigated hosted x86_64 GitHub Actions variance, but that slowed the feedback loop and mixed local implementation work with hosted-runner noise.

Recent GitHub Actions run `25212031553` on `ubuntu-24.04` x86_64, commit `f6262674d44d6fde241c60888e68c575993f5ab1`, reported:

- Zig `insert_reserved`: 40.122 ns/op
- C++ `insert_reserved`: 30.609 ns/op
- Gap: Zig is 31.1% slower

Local aarch64 median results from the Goal 0003 final audit reported:

- Zig `insert_reserved`: 25.472 ns/op
- C++ `insert_reserved`: 25.779 ns/op
- Gap: Zig is 1.2% faster

The current scope is now local aarch64 development-host performance. GitHub Actions benchmark output must not be used as a Goal 0004 pass/fail gate.

## Objective

Improve Zig `insert_reserved` performance so it is at least on-par with C++ `parallel-hashmap` on:

- the local development host, currently aarch64

Primary target:

- `insert_reserved`: Zig must be no slower than C++ on the same local machine and benchmark run, using repeated local samples and medians.

Secondary targets:

- Do not regress `lookup_hit`, `lookup_miss`, `iterate`, `mixed`, `remove`, `string_insert`, `string_lookup`, `high_load_miss`, or `tombstone_churn` by more than 10% from the Goal 0003 final audit unless the regression is documented and justified.
- Preserve all public APIs.
- Preserve allocation-failure correctness.
- Preserve node-container pointer stability.
- Keep the implementation portable; do not add micro-optimizations that only improve one target such as x86_64.

## Baseline To Beat

Use the existing Zig and C++ benchmark workloads unless this goal deliberately fixes benchmark fairness. If benchmark code changes, record old and new numbers.

Historical GitHub Actions x86_64 baseline from run `25212031553`, for context only:

- Zig `insert_reserved`: 40.122 ns/op
- C++ `insert_reserved`: 30.609 ns/op
- Zig gap: 31.1% slower

Local aarch64 baseline from Goal 0003 final audit:

- Zig `insert_reserved`: 25.472 ns/op
- C++ `insert_reserved`: 25.779 ns/op
- Zig gap: 1.2% faster

Required final threshold:

- Zig `insert_reserved` <= C++ `insert_reserved` on local benchmark runs.

Use multiple benchmark runs locally and compare medians. Do not claim parity from a single noisy outlier or from a hosted CI run.

## Required Investigation

Before changing code, inspect the current Zig reserved-insert path and the upstream C++ equivalent.

Inspect the Zig implementation for:

- `put`, `insert`, `getOrPut`, and the internal insertion helper used by `insert_reserved`
- hash splitting into H1/H2
- probe-sequence initialization and advancement
- first non-full-slot search
- equality checks during insertion
- growth-left accounting
- tombstone reuse
- control-byte writes and cloned-control-byte maintenance
- slot initialization and value movement
- any extra safety checks or branches still present in ReleaseFast

Inspect upstream C++ from:

- `.deps/parallel-hashmap-2.0.0/parallel_hashmap/phmap.h`

Focus on:

- `raw_hash_set::emplace`
- `find_or_prepare_insert`
- `prepare_insert`
- `find_first_non_full`
- control-byte writes
- growth-left accounting
- resize/tombstone cleanup policy

Record source-inspection findings in `checkpoints.md` before making performance changes.

## Optimization Direction

Focus on production hot-path improvements, not benchmark-only shortcuts.

Likely areas to evaluate:

- Avoid duplicated probe work between lookup and insertion.
- Avoid unnecessary equality checks when the first probed group has an empty slot.
- Reduce branches in the reserved-insert fast path.
- Separate the no-grow reserved-insert path from the general grow-capable insert path if it measurably helps.
- Keep `growth_left` and tombstone accounting branch-light.
- Ensure H2/control-byte writes and cloned-control-byte updates are minimal and correctly ordered.
- Check whether slot initialization performs avoidable copies or default initialization.
- Evaluate shared portable probe and insertion-path code before considering target-specific backend work.
- Use targeted inline hints only when measurements show a real improvement.

Do not reduce correctness checks, remove tests, specialize behavior only for benchmark key types, or add target-only micro-optimizations that compromise the library's portable design.
